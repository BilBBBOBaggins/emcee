# ADR-014: Фиксы консистентности промпт-канона (разбор внешнего вердикта)

Дата: 2026-06-30 · Статус: Принят

> Решение принято прогоном адверсивной панели (red-team → blue-team → arbiter), см. [core/adversarial-panel.md](../../core/adversarial-panel.md).

## Коротко

Внешний ревьюер предложил «разжать промпты — они слишком жёсткий runtime prompt» (семь пунктов).
Адверсивная панель плюс реальный прогон субагентов постановила: **обе крайние рамки неверны.**
«Разжать» — категориальная ошибка: аппаратный tool-scoping (ограничение набора инструментов на уровне
рантайма) — несущая гарантия пакета, открывать скоуп нельзя. «Всё by-design» (первая линия защиты) — тоже
не выжила: канон местами **сам нарушает свой принцип** (требует от ролей без Bash того, что скоуп им отнял)
и содержит **невыполненный пункт ADR-001**. Решение — **между ними**: примерно 15–18 точечных правок
документации, которые приводят канон в согласие с его же несущим принципом. Ноль инженерных месяцев,
ноль изменений в наборах инструментов (`tools:`-скоупе) ролей.

Самое ценное: эмпирический прогон субагентов **опроверг** вывод, на котором сошлось чтение, — будто
architect не может посчитать метрики без Bash (может, через делегацию задачи), а reviewer молча фабрикует
результат (нет, честно флагует пробел). Эмпирика важнее согласованного чтения.

## Context

`emcee` использует `tools:`-скоупинг в `.claude/agents/*.md` как **аппаратную** границу ролей (reviewer
read-only, architect docs-only и т.д.) — это дифференциатор пакета (`core/portability.md`). North star:
качество > токенов; строгость сознательна. Внешний вердикт предложил смягчить строгость. Проверка вердикта
важна, т.к. канон копируется в каждый проект — дефект тиражируется.

Панель прогнала семь пунктов через red-team, blue-team и arbiter (с независимым Bash-фактчеком), затем
закрыла открытое стоп-условие реальным прогоном reviewer- и architect-субагентов.

## Decision

Рамка: **скоуп защищать, но канон обязан перестать требовать от ролей без Bash того, что скоуп у них
отнял.** Пакет правок (все — правки промптов и документации, файлы и строки верифицированы):

1. **Handoff exit→dispatch.** `roles/developer.md` — exit-отчёт обязан содержать **список фактически
   изменённых файлов**; `.claude/commands/role.md` и `core/task-protocol.md` — exit-отчёт предыдущей роли
   прокидывается следующей (на solo — пользователю или оркестратору).
2. **Reviewer ↔ diff (сильнейшее попадание).** `core/pipeline.md` — вход reviewer = «exit-отчёт со списком
   изменённых файлов», НЕ «готовый diff»; `roles/reviewer.md` — дисклеймер «фактический git-diff в роли не
   верифицируется (read-only by hardware)». Чинит нарушение собственного принципа в каноне.
3. **arbiter и Bash.** `.claude/agents/arbiter.md` (есть Bash) прав; устаревшие `.claude/README.md` **и
   строка матрицы `core/portability.md`** привести к «arbiter имеет Bash для узкого codex-фактчека».
4. **Исполнить binding-пункт ADR-001.** Вынести codex-команду в один источник (`core/second-model.md`) с
   плейсхолдером модели; синхронизировать фолбэк `.claude/commands/panel.md` («codex в любом случае») с
   `adversarial-panel.md` и ADR-001 («рекомендован, не обязателен»); убрать ссылку на приватный
   `~/.claude/CLAUDE.md`.
5. **Легализовать «[нужны данные: X]».** `core/principles.md` — недостающий ВНЕШНИЙ факт (нет логов/CI/
   доступа) → назвать факт и путь его получения; это НЕ угадывание. Анти-угадывание сохраняется.
6. **QA UAT.** `roles/qa-uat.md` — встроить в priority-блок: явное business-rule от BA/SA перебивает common
   sense; ожидания сверх BA/SA — это recommendation, не молчаливый acceptance.
7. **Рамка строгого правила** — заменить «строгость = фича» четырёхчастным тестом (входной артефакт /
   владелец / exception-path / дешёвая верификация). Мелочь: `roles/developer.md` «перед каждым действием»
   → «перед каждым **значимым** действием».

**Architect-радиус (поправлен эмпирикой):** `roles/architect.md` — благословить делегацию сбора метрик
**read-only-замерщику** (Bash только git/test/wc, НЕ write — иначе размывается docs-only); нет делегации
или метрик → пометка «[метрики не получены]», не глазомер.

## Consequences

**Положительные:** канон приходит в согласие с собственным несущим принципом; устранён невыполненный
binding-пункт ADR-001; неисполнимый класс (дебаг без доступа) легализован; reviewer-гарантия сужена до
честно-исполнимой. Дёшево и обратимо.

**Цена и остаточные риски:**

- Зазор «самодекларация developer'а ≠ verified diff» необнуляем. Эмпирический прогон опроверг «структурную
  невидимость»: reviewer через `Glob` перечисляет дерево файлов и **находит** даже незаявленный файл
  (в тесте поймал `config.json` с live-секретом). Реальный предел — **change-attribution**: без diff нельзя
  надёжно отличить *изменённый* файл от пред-существующего; в маленьком проекте обзор дерева ловит лишнее
  случайно, в **большом репо** незаявленное изменение без ссылки практически пропускается. Правка №2 делает
  предел видимым (дисклеймер). Полное закрытие — через диспетчер с Bash, который подаёт реальный `git diff`,
  без открытия скоупа reviewer'а.
- «docs-only **аппаратно**» для архитектора (и любого холдера делегируемых задач) — на деле **мягкая**
  гарантия: делегированная задача может породить write-способного потомка. Правка architect-радиуса
  ограничивает делегата read-only, но матрица `portability.md` в целом завышает «аппаратность» для
  таких ролей — нужна отдельная ревизия формулировок матрицы.
- Дешёвая проверка: подтвердить слепоту reviewer'а к не-связанным файлам прогоном (промпт = `A.ts`,
  факт = несвязанный `config.json`). Ожидаемо подтвердится; правка №2 закрывает.
- Не строить тяжёлый verified-diff-протокол. Диспетчер `/role` исполняется в main-сессии (Bash есть) → он
  сам считает `git diff --name-only` и подаёт reviewer как **авторитетный** change-set, кросс-чекая
  самодекларацию developer'а. Это закрывает change-attribution в репо любого размера, **не давая reviewer
  Bash** (скоуп цел). Прозовый floor без Bash-диспетчера: самодекларация + `Glob` + дисклеймер. Публичную
  гарантию формулировать точно: «reviewer read-only by hardware (код не запускает); проверяет переданный
  change-set (на Claude Code — реальный git diff от диспетчера; в прозовом режиме — самодекларация
  developer'а + обзор дерева); независимую верификацию рабочего дерева НЕ заявляет». Не продавать
  «ревьюит diff» как независимую проверку.
- Сами ~15–18 правок канона **не применены** этим ADR — он фиксирует решение. Применение — отдельный шаг
  под подтверждение владельца (границы агентности: правки канона тиражируются в каждый проект).

## Alternatives considered

- **«Разжать промпты» (вердикт ревьюера)** — отклонено: открытие скоупа уничтожило бы read-only/docs-only
  гарантии, несущий дифференциатор пакета. Red сам признал направление «скоуп защищать» верным.
- **«Всё by-design, только doc-sync» (исходная версия)** — отклонено: фактически ошибочно (`pipeline.md`
  сам требует diff у reviewer'а без Bash) и опиралось на полу-существующий handoff; north star
  использовался как нефальсифицируемый щит против валидной критики.
- **Дать reviewer/architect узкий read-only Bash** — отклонено: ломает аппаратную read-only/docs-only
  гарантию. Делегация задачи (architect) и handoff-список (reviewer) решают задачу без открытия скоупа.
