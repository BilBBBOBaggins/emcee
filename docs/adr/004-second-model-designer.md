# ADR-004: Вторая модель всем ролям (узко, opt-in) + дормантная роль Designer

Date: 2026-06-27
Status: Proposed

> Решение принято прогоном адверсивной панели (red-team → blue-team → arbiter), см. [core/adversarial-panel.md](../../core/adversarial-panel.md). Применяет нормы [ADR-001](001-scope-process-overlay.md) (пакет не владеет owned-долгом) и [ADR-003](003-first-km-intake.md) (активация новой возможности — под гейтом ретро оператора).

## Коротко

Оператор запросил две фичи. Первая — дать вторую модель (codex) всем ролям, не только панели и C+:
сделано как один core-док `second-model.md` с узкими opt-in триггерами (не строка в восьми файлах
ролей). Вторая — роль Designer, рисующая макет: построена **дормантно** и **только текстом** —
Designer выдаёт вайрфрейм как код прямо из спеки, а растровый макет остаётся эфемерным скретчем «для
глаз» и в git не попадает. Растровую часть как источник кода (image-gen → vision → код) и PNG как
версионируемый артефакт выбросили: ненадёжно и противоречит охвату пакета.

## Контекст

Оператор (соло, ground truth) запросил две фичи:

- **(A)** дать **вторую модель (codex) всем ролям**, а не только панели и C+.
- **(B)** роль **Designer**, которая рисует макет через codex и переводит картинку в вайрфрейм, с
  версионированием и PNG-макета, и вайрфрейм-кода.

Премисса проверена (web 2026): codex CLI реально генерирует изображения (gpt-image-2) и принимает их
на вход (vision), так что пайплайн технически возможен. Но v1 в исходном виде столкнулся с нормами
пакета.

## Решение

**Строим узко и только текстом; растровую часть выбрасываем; активацию Designer — под гейт.**

**A. Вторая модель — `core/second-model.md`** (построено сейчас):

- Один core-док с **узкими opt-in триггерами** «когда любая роль зовёт codex вторым взглядом» на
  high-stakes выводе (reviewer на финдингах, architect перед ADR, SA/BA на требованиях, debugger на
  гипотезе). Это **не** строка в восьми файлах `roles/*.md` — такой touchpoint был бы мёртвым долгом.
- **opt-in, не обязательно.** «вторая модель (codex) на каждом шаге» отклонён: это ритуал + упор в
  лимиты квоты второй модели.
- **Честная оговорка:** codex — это code/reasoning-модель; как ревьюер дизайна или требований она
  слабее, поэтому на не-коде её вызов помечать и замерять пользу (в PROCESS-METRICS).
- **Не ослабляет локальные протоколы:** если запущена панель или C+, действует их собственное
  правило про codex; общий second-model — отдельный opt-in для обычных ролей.

**B. `roles/designer.md` — дормантный регламент** (построено сейчас, ноль runtime):

- Метод записан в markdown, но роли **нет в `roles.json`** → нет цифры, числовой пайплайн её не
  вызывает. Ad-hoc прямой промпт — это ручное чтение метода, а не активация.
- Выход — **вайрфрейм как КОД** прямо из спеки (HTML / SVG / скелет): версионируемый, dev-ready,
  диффится.
- **Картинка-макет — эфемерная опция «для глаз»** в заранее игнорируемом пути (`scratchpad/design/`).
  Она не коммитится и не является источником кода. Правила в `.gitignore` нет специально — поэтому
  Designer сначала предлагает добавить его оператору. Обратного чтения «PNG → код» не делается.
- Designer-как-роль-инструкция в репозитории пользователя — это делегирование (как day-0 делегирует
  init); пакет не владеет дизайн-артефактами → ADR-001 цел.

## Что выброшено (признано панелью фатальным)

- image-gen → PNG → vision → код **как источник кода** (растровый dev-контракт ненадёжен:
  Design2Code / DCGen галлюцинируют layout).
- PNG-макет как **канонический версионируемый артефакт** + UI-регрессия в репозитории (бинарь
  противоречит ADR-001: не диффится, протухает).
- Строка «зови codex» в восьми ролях; обязательный проход codex.

## Последствия

**Плюсы:** вторая пара глаз обобщена на весь пайплайн без touchpoint-долга; Designer доступен как
метод без owned-долга и без бинаря в git; нормы ADR-001/002/003 целы; цифры ролей 0–7 не тронуты.

**Риски и открытые вопросы:**

- [ ] **O1-D — стоп-гейт активации Designer.** Активацию Designer (цифра в `roles.json`, место в
      `task-protocol.md`, бутстрап генератора, selftest-инвариант) делать только после ретро 2–3
      реальных UI-стартов, показавшего потерю на «нет вайрфрейма». Не было потери → не активировать.
- [ ] Пользу второй модели (codex) на НЕ-коде (SA/BA/дизайн) логировать в `docs/PROCESS-METRICS.md`.
      Если за 2–3 фичи она не ловит класс дефектов — сузить триггеры обратно к reviewer/architect/C+.
- [ ] Вторая модель (codex) работает на ограниченной квоте; при росте частоты вызовов следить за
      rate-limit (узкие триггеры — первичная защита).

## Рассмотренные альтернативы

- **v1 (codex всем обязательно + Designer с PNG-каноном + vision raster → код).** Отклонён:
  обязательность = ритуал + rate-limit; PNG-канон противоречит ADR-001; raster → код ненадёжен;
  активация без гейта противоречит ADR-003.
- **codex точечно только в reviewer/architect.** Поглощён узкими триггерами `second-model.md` (те же
  места + SA/BA/debugger под замер) — это не отдельная сущность.
- **Designer, выдающий только вайрфрейм-код без картинки.** По сути и принят; картинка оставлена как
  эфемерная опция «для глаз», а не как версионируемый артефакт.
