## По ситуации

Это **роутер скиллов и регламентов**: ситуация слева → готовый скилл/файл справа. Совпала ситуация —
**вызови существующий скилл, не переизобретай** его инлайн (дебаг по наитию вместо `debugging`,
самодельный чек качества вместо `code-quality`). Скилл уже инкапсулирует метод; реизобретение теряет
его гарантии и жжёт контекст. Не уверен, какой скилл — спроси пользователя, не угадывай.

- Дебаг чего-то сломанного → [core/debugging.md](core/debugging.md)
- Вопросы о качестве кода → [core/code-quality.md](core/code-quality.md)

## Testing philosophy

{{Выбери один из трёх паттернов и оставь только его секцию. Остальные удали.}}

### Вариант 1: Outside-in BDD (для B2B с domain expertise)

Pipeline формирования тестов и кода:

1. Системный аналитик или BA формирует acceptance criteria на языке домена (Given/When/Then).
2. QA UAT превращает criteria в формальные тест-кейсы с ожидаемым видимым поведением.
3. QA E2E или Developer пишет код тестов.
4. Developer реализует код чтобы тесты проходили.

Тесты — **спецификация**, не проверка. Тест красный = баг в коде (или неполная имплементация), не проблема теста.

Применять для: B2B продуктов, регулируемых доменов, продуктов с внешними domain experts.

### Вариант 2: Test-along (для индивидуальной разработки)

Агент пишет код и тесты в рамках одной задачи одновременно.

- Тесты покрывают критичные paths и edge cases
- Не гонимся за 100% coverage
- Unit-тесты для бизнес-логики обязательны
- Integration-тесты — для модулей на стыках

Применять для: solo-разработки, быстрой итерации, когда domain-эксперт не в команде.

### Вариант 3: Classic TDD (для чётко определённых компонентов)

Строгий red-green-refactor цикл применяется избирательно к модулям где контракт предельно ясен:

- Парсеры форматов
- Вычислительные функции
- Трансформации данных
- Валидаторы

Для остальной разработки используется Test-along.

Применять для: библиотек, SDK, компонентов с жёстким контрактом.

## Документация: человекочитаемость

Доку пишем под аудиторию, и для разных аудиторий — по-разному:

- **Человеко-facing** (README, QUICKSTART, руководства, ADR) — **самодостаточны и человекочитаемы**:
  не предполагают, что прочитан весь набор `.md`; аббревиатуры расшифровываются при первом упоминании;
  длинные вложенные скобки разбиваются в предложения, списки или таблицы; у ADR в начале — раздел
  «Коротко» с сутью решения.
- **Агенто-facing** (`core/`, `roles/`, этот входной файл регламента, обвязка рантайма `.claude/`/`.codex/`) —
  **плотность сознательна**: она экономит контекст агента. Не «упрощать» их под человека.

Правишь человеко-facing доку — оставляешь её человекочитаемой; заводишь новый ADR — с разделом «Коротко».

## Эволюция этого документа

Этот входной файл регламента — живой. Правила добавляются когда агент делает ошибку которую можно предотвратить формализацией. Правила удаляются когда они over-specialized под ушедший контекст. Раз в 1-3 месяца — review всего документа, удаление неактуального.

Если ты как пользователь заметил что агент делает одну и ту же ошибку в разных задачах — это сигнал что во входном файле регламента нужно новое правило. Добавь его в соответствующий раздел.
