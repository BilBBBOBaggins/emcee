# Master of Ceremonies
Регламент агентной разработки на **Claude Code и Codex** — основную работу делает агент, а не человек. Короткое имя пакета и репозитория — `emcee`.

## Что это такое

Это **слой регламента работы агента** поверх вашего проекта — нового или уже существующего.
«Регламент» здесь — это набор правил, по которым агент пишет код: чёткие роли (архитектор,
разработчик, ревьюер и т.д.), формат постановки задач, числовые команды для запуска ролей
и процедура состязательной проверки важных решений.

Пакет даёт отработанные правила, структуру задач и роли. **Само приложение он не создаёт.**
Код, `go.mod` / `package.json`, Makefile, CI вы инициализируете обычным инструментом
(`go mod init`, `create-next-app`, `uv init` и т.п.) — или поручаете это самому агенту. То есть
пакет — это быстрый старт *регламента*, а не *приложения*.

Почему так, а не иначе, зафиксировано в [ADR-001](docs/adr/001-scope-process-overlay.md):
владеть устаревающим scaffold'ом под десятки стеков непосильно одному человеку, а агент и так
умеет инициализировать стек одной командой.

### Claude Code default, Codex built-in

Нативный рантайм по умолчанию — **Claude Code** (CLI/расширение от Anthropic): авточтение
`CLAUDE.md`, каталог `.claude/` (субагенты, скиллы, slash-команды, хуки), plan mode. **Codex CLI —
встроенная цель, не ручной перенос:** генерируйте проект с флагом `--harness codex`, и пакет
кладёт нативную для Codex обвязку — вход `AGENTS.md` (Codex авто-читает; файла `CLAUDE.md` в
codex-проекте нет), агент-профили `.codex/agents/*.toml`. Метод (роли, гейты, пайплайн) — общий:
он живёт в нейтральном ядре `core/`, на которое указывают оба входа. Как это устроено —
[ADR-011](docs/adr/011-process-layer-and-multimodel-build.md) (мультимодельная сборка) и
[ADR-012](docs/adr/012-entry-file-per-harness.md) (per-harness вход).

Где честная граница: часть гейтов на Codex **деградирует из mechanical в accountability** (хуки в
headless `codex exec` не срабатывают — для жёсткого энфорсмента CI/pre-commit); это явно
размечено. Перенос на **прочие** рантаймы (Cursor, свой harness), для которых встроенной цели нет —
по-прежнему ручная работа по карте: что завязано на рантайм и как переносить —
[core/portability.md](core/portability.md); почему граница в прозе, а не физические слои —
[ADR-009](docs/adr/009-portability-boundary.md). Фундаментальные приёмы (контракт-первый цикл,
декомпозиция задач, отделение разведки от производства, адверсивный метод) от рантайма не зависят.

## С чего начать

**За 5 минут — [QUICKSTART.md](QUICKSTART.md).** Там пошагово разобраны три сценария:

- **новый проект** (пустой каталог);
- **существующий проект без нашего регламента** (накладываем рядом с кодом);
- **существующий проект со старым регламентом** (нужно обновить).

Плюс ежедневный рабочий цикл и то, как держать дисциплину. Ниже на этой странице — справка
по самому пакету: его устройство, содержимое папок и опциональная обвязка.

## North star: качество результата важнее экономии токенов

Главный принцип пакета — **качество результата важнее экономии токенов и времени.** Через эту
оптику стоит читать все рекомендации:

- Дисциплина чистого контекста (тонкие индексы, ленивая подгрузка файлов) нужна не ради
  дешевизны, а ради **качества рассуждения** агента: меньше мусора в контексте — точнее ответ.
- Там, где качество требует больше работы — вторая модель в адверсивной панели, проходы
  верификации, независимый автор тестов — это **поведение по умолчанию**, а не «если хватит
  бюджета».

## Два режима использования

У генератора `new-project.py` есть флаг `--mode` с двумя значениями.

- **`new`** — кикстарт нового проекта в пустом каталоге. Генерирует каркас регламента, а дальше
  агент сам инициализирует стек по командам init.
- **`overlay`** — наложить регламент на **существующий** проект, ничего не затирая. Ваши
  `README.md`, код и собственный `CLAUDE.md` остаются нетронутыми.

У режима `overlay` два под-случая: проект **без** нашего регламента и проект **с устаревшим**
регламентом (его нужно обновить — см. трек А2 в [QUICKSTART.md](QUICKSTART.md)).

## Как агент с этим работает

Claude Code автоматически читает файл `CLAUDE.md` в корне проекта в начале каждой сессии. Оттуда
он видит ссылки на остальные файлы регламента и переходит к ним по мере необходимости.

В `CLAUDE.md` определены **числовые команды** — короткий способ запустить нужную роль:

| Команда | Что значит |
|---|---|
| `N` (одно число) | Архитектор входит в день N: читает проект, выдаёт статус. |
| `R D` (два числа) | Роль R входит в контекст дня D без конкретной задачи (review, планирование). |
| `R D T` (три числа) | Роль R берёт задачу T из гайда дня D. **Это основной режим работы.** |

Здесь **«день»** — это инкремент плана проекта. Задачи дня лежат в файле `docs/day-<N>-guide.md`,
и каждая задача содержит готовый блок «Промпт для Claude Code», секцию «После выполнения» и
«Коммит». Полный рабочий пример дня, ролей и связки артефактов (spec → scenarios → test-cases →
код) лежит в [examples/](examples/).

Исполняемая обвязка (субагенты, slash-команды, хуки) **опциональна** — см. раздел
[«Опциональная обвязка»](#опциональная-обвязка-claude-code) ниже. Без неё пакет работает в простом
«прозовом» режиме: агент читает markdown и становится нужной ролью.

## Структура репозитория

~~~
emcee/
├── new-project.py                  # генератор: собирает регламент (--mode new|overlay)
├── sync-roles.py                   # перегенерирует таблицы ролей из roles.json (--check ловит рассинхрон)
├── regimen-doctor.py               # проверка готовности регламента в проекте; копируется в проект
├── render-handbook.py              # собирает HTML: handbook.html (справочник) + quickstart.html
├── selftest.py                     # самотест: проверки генератора и инвариантов пакета
├── roles.json                      # ЕДИНСТВЕННЫЙ источник карты ролей (цифра → роль)
├── CLAUDE.md                       # точка входа, адаптируется под проект
├── QUICKSTART.md                   # пошаговый старт (НЕ копируется в проект)
├── docs/adr/                       # решения о самом пакете (НЕ копируется в проект)
├── .claude/                        # опциональная обвязка: agents/ skills/ commands/ hooks/
├── core/                           # базовые правила агента (обычно не редактируются)
├── stack/                          # правила под язык/фреймворк — выбери нужные, остальные удали
├── architecture/                   # архитектурные паттерны — выбери применимые
├── domain/                         # доменные паттерны — выбери свой
├── roles/                          # определения ролей — обычно нужны все
└── examples/                       # один сквозной заполненный пример (НЕ копируется в проект)
~~~

Эти три источника **не копируются** в генерируемый проект, потому что они описывают сам пакет, а не ваш
код: каталоги `docs/adr/` (решения о пакете) и `examples/` (демонстрация) плюс файл `QUICKSTART.md`.

### Скрипты-инструменты

- **`new-project.py`** — генератор регламента. Подробно — в разделе [«Как пользоваться»](#как-пользоваться) ниже.
- **`sync-roles.py`** — перегенерирует таблицы ролей из `roles.json`. Флаг `--check` ловит рассинхрон.
- **`regimen-doctor.py`** — гейт готовности регламента в вашем проекте: проверяет незаполненные
  плейсхолдеры, висячие ссылки, синхронность ролей, настройки и команды. Копируется в проект.
- **`render-handbook.py`** — собирает из markdown-источников два HTML-файла: `handbook.html`
  (полный справочник) и `quickstart.html` (онбординг). HTML не коммитится — источник истины это
  сами `.md`; HTML регенерируется командой `python3 render-handbook.py`.
- **`selftest.py`** — самотест: проверки генератора и инвариантов пакета (число растёт по мере
  добавления инвариантов; актуальное — в выводе `python3 selftest.py`).

## Как пользоваться

### Быстрый старт: генератор (рекомендуется)

Полный пошаговый сценарий «с нуля до первой задачи» (оба режима + дисциплина) — в
**[QUICKSTART.md](QUICKSTART.md)**. Здесь — справка по самому генератору.

`new-project.py` собирает регламент: копирует `core/` и `roles/`, добавляет выбранные модули
`stack` / `architecture` / `domain`, заполняет что может в `CLAUDE.md`, создаёт заготовки в `docs/`
и, опционально, кладёт обвязку `.claude/`.

Чем отличаются два режима `--mode`:

- **`new`** (по умолчанию) — кикстарт нового проекта в пустом каталоге. Дополнительно создаёт
  `docs/day-0-guide.md` — гайд инициализации, который **делегирует** работу агенту: тот сам
  запускает `go mod init` / `create-next-app` / `uv init` под выбранный стек (в актуальной версии
  инструмента, а не из зашитого scaffold'а), получает зелёный baseline и только потом берёт День 1.
- **`overlay`** — наложить регламент на существующий проект, ничего не затирая. Если `CLAUDE.md`
  уже есть, регламент кладётся рядом как `CLAUDE.regimen.md` для ручного слияния; `day-0-guide` не
  создаётся (проект уже инициализирован).

~~~bash
./new-project.py --list                      # доступные модули
./new-project.py                             # интерактивно (спросит режим)

# кикстарт нового проекта:
./new-project.py --name "Acme Teams" --dir ../acme --mode new \
    --backend go --frontend react-nextjs \
    --arch modular-monolith,multi-tenant --domain b2b-saas \
    --testing test-along --wiring yes

# наложить регламент на существующий проект:
./new-project.py --name "Acme Teams" --dir ../existing-repo --mode overlay \
    --backend go --testing test-along --wiring yes
~~~

После генерации скрипт печатает чеклист: незаполненные `{{...}}`, висячие ссылки на невыбранные
модули, в режиме `overlay` — список нетронутых существующих файлов, и следующие шаги.

**А если вашего стека ещё нет в `stack/`?** Укажите любое имя (`--backend rust`) — генератор
создаст **скелет** `stack/rust.md` из шаблона [stack/_TEMPLATE.md](stack/_TEMPLATE.md). В скелете
будут все нужные секции, включая обязательный раздел «Чистая сборка», на который ссылается
[core/quality-gates.md](core/quality-gates.md). Заодно генератор выдаст готовый промпт, которым
Claude Code дозаполнит этот файл. Новые `architecture/` и `domain/` — это просто свободные
`.md`-файлы без жёсткого контракта: кладёте в папку, и всё.

### Вручную (то же, что делает генератор)

1. Скопируйте **содержимое** `emcee/` в корень нового проекта так, чтобы `CLAUDE.md`
   оказался в корне репозитория, а папки `core/ stack/ architecture/ domain/ roles/` лежали рядом с
   ним на верхнем уровне. Claude Code авто-читает именно корневой `CLAUDE.md`, и относительные
   ссылки внутри него рассчитаны на этот layout.

   **Не переносите** `CLAUDE.md` внутрь `.claude/` — оттуда он не подхватывается автоматически, и
   все ссылки поедут. Каталог `.claude/` зарезервирован под исполняемую обвязку — см.
   [«Опциональная обвязка»](#опциональная-обвязка-claude-code).

2. Откройте `CLAUDE.md` и заполните плейсхолдеры `{{...}}` под свой проект.

3. Заполните оставшиеся `{{...}}` **и в тех core/role/arch-файлах, которые оставили** — не только в
   `CLAUDE.md`. Найти все:

   ~~~bash
   grep -rn '{{' . --include='*.md'
   ~~~

   Особое внимание — `roles/qa-e2e.md` и `roles/qa-uat.md` (по ~8 плейсхолдеров каждый): без них
   роли нерабочие.

4. Удалите из `stack/`, `architecture/`, `domain/` файлы, которые не применимы. После удаления
   проверьте, что не осталось висячих ссылок:

   ~~~bash
   grep -rn 'имя-удалённого-файла' . --include='*.md'
   ~~~

   (Ссылки из `core/` на `architecture/` и `domain/` сделаны условными — см.
   [core/code-quality.md](core/code-quality.md).)

5. Подкорректируйте `roles/`, если нужны свои роли или другие цифры. **Карта цифр — единственный
   источник в `roles.json`:** правите его → запускаете `python3 sync-roles.py` (он перегенерит
   таблицы в `CLAUDE.md` и `.claude/commands/role.md`; `--check` ловит рассинхрон).

6. Закоммитьте как стартовый commit проекта.

## Содержание папок

### core/ — базовые правила агента

Фундаментальные принципы работы агента, независимые от стека и типа проекта. Файлы делятся на две
группы по тому, когда они читаются.

**Читаются в начале каждой сессии** (они перечислены в `CLAUDE.md` → «Обязательно читать»):

- `pipeline.md` — **сквозной нарратив «как работать»**: старт проекта (`/kickoff`) → онгоинг
  (`R D T`), кто что делает, откуда берутся гайды дня, пример со всеми ролями.
- `principles.md` — факт vs гипотеза, минимальный контекст, видимость работы.
- `task-protocol.md` — как агент получает и выполняет задачи.
- `quality-gates.md` — критерии завершённости задачи, пороги-сигналы на размер файлов, правила тестов.
- `constitution.md` — несущие non-negotiable правила + протокол сверки preflight/exit.

**Читаются по необходимости** (их подтягивают роли и `CLAUDE.md` → «По ситуации» — это сознательное
разделение, чтобы не грузить контекст лишним):

- `debugging.md` — как правильно дебажить: одновременный сбор логов со всех слоёв.
- `code-quality.md` — стандарты кода: SRP, именование, безопасность, читаемость.
- `memory.md` — память между сессиями: иерархия `CLAUDE.md` + auto-memory, дисциплина (<200 строк
  на файл памяти, тонкий индекс + ленивые topic-файлы).
- `spec-driven.md` — цикл «контракт-первый» (C+) для задач с жёстким контрактом: тест пишется до
  кода, независимым автором, с состязательной вычиткой тестов ([ADR-002](docs/adr/002-spec-driven-cplus.md)).
- `adversarial-panel.md` — состязательный разбор арх-решения до коммита: red-team бьёт по решению
  (и привлекает codex как вторую модель), blue-team честно защищает, arbiter выносит обязывающий
  вердикт, дальше синтез v2 и ADR. Запуск — командой `/panel`.
- `second-model.md` — codex как вторая пара глаз на важном выводе любой роли (opt-in, узкие
  триггеры) — обобщает то, что в панели и C+ уже зашито ([ADR-004](docs/adr/004-second-model-designer.md)).
- `portability.md` — граница переносимости: что завязано на рантайм Claude Code, а что переносится
  как идея; нотация `origin:` и карта harness-зависимостей для будущего форка на другой рантайм
  ([ADR-009](docs/adr/009-portability-boundary.md)).

### stack/ — правила под язык и фреймворк

- `go.md` — Go 1.23+ (Echo, sqlc, slog, testify).
- `python.md` — Python 3.12+ (uv, ruff, mypy strict, FastAPI, pytest).
- `react-nextjs.md` — Next.js App Router, TypeScript strict, shadcn/ui, TanStack Query.

Для своих стеков добавляйте новые файлы (см. `stack/_TEMPLATE.md`).

### architecture/ — архитектурные паттерны

Выбирайте применимые под проект, остальные удаляйте.

**Композиция (ключевой выбор):**
- `layered-architecture.md` — общий паттерн слоёв с однонаправленными зависимостями.
- `modular-monolith.md` — один деплой, чёткие модули внутри.
- `microservices.md` — независимо деплоящиеся сервисы.

**Изоляция данных:**
- `multi-tenant.md` — multi-tenancy с RLS в Postgres (для B2B).

**UI-архитектура:**
- `three-tier-with-bridge.md` — для desktop/mobile-приложений с native + declarative UI.

**Коммуникация:**
- `event-driven.md` — асинхронное взаимодействие через события.

**AI-специфика:**
- `ai-heavy.md` — LLM-инфраструктура, версионирование промптов, eval suite, RAG.
- `agentic-workflows.md` — multi-agent системы, память, использование инструментов.

**Тестирование:**
- `autonomous-testing.md` — паттерн TestDriver для UI-heavy приложений.

### domain/ — доменные паттерны

- `b2b-saas.md` — B2B SaaS: onboarding, SSO, billing, audit log, customer success.
- `regulated.md` — compliance, персональные данные, ФЗ-152, GDPR, аудит, инциденты.

### roles/ — роли для agentic workflow

**Архитектор (lead)** — `architect.md`: системное проектирование, ADR, статус дня. Вызывается одним
числом `N` (вход в день) или `/kickoff`. **Цифры в `roles.json` у него нет** — он не участник числового
пайплайна `R D T`, а его ведущий (см. [ADR-007](docs/adr/007-kickoff-pipeline.md), [core/pipeline.md](core/pipeline.md)).

Активные числовые роли (единственный источник — `roles.json`, цифры 0–7):

- `reviewer.md` (0) — code review, документирование проблем (не исправляет).
- `developer.md` (1) — основной кодинг-агент.
- `qa-e2e.md` (2) — E2E-тестирование полного стека.
- `ba.md` (3) — бизнес-аналитик, извлечение сценариев из кода.
- `qa-uat.md` (4) — написание тест-кейсов для заказчика.
- `sa.md` (5) — системный аналитик, опрос domain experts.
- `debugger.md` (6) — реактивная роль для конкретных багов.
- `devops.md` (7) — CI/CD, pre-commit гейты, секреты, деплой (мост от локального вывода к проду).

**Дормантные роли** — метод записан, но в `roles.json` их нет: цифры у них нет, в числовом
пайплайне они не участвуют. Доступны ad-hoc (по имени), а активация цифрой — под гейтом реального
опыта (см. ADR ниже):

- `designer.md` — для UI-фич выдаёт вайрфрейм **как код** прямо из спеки. Картинка-макет —
  эфемерный скретч в `scratchpad/design/`, в git не коммитится ([ADR-004](docs/adr/004-second-model-designer.md)).
- `auditor.md` — холистический read-only аудит здоровья всего проекта + карта больных мест. Ловит
  межзадачный архитектурный дрейф, который не видят per-task reviewer и статус архитектора
  ([ADR-005](docs/adr/005-auditor-role.md)).
- `upgrader.md` — обновление отставшего регламента: агент по git-диффу делает report-first,
  авто-обновляет только чистые package-owned файлы, а человек ревьюит и коммитит
  ([ADR-006](docs/adr/006-regimen-upgrade.md)).

### examples/ — один сквозной пример

Полностью заполненный пример одной фичи («пригласить участника» в B2B SaaS на Go + Next.js). Он
показывает **формат** артефактов, на которые ссылаются роли: заполненный `CLAUDE.md`, гайд дня с
блоком «Промпт для Claude Code», статус-файл, связку `spec → scenarios → test-cases`, ADR. В
реальный проект не копируется — это демонстрация. Подробнее — [examples/README.md](examples/README.md).

## Опциональная обвязка Claude Code

Пакет работает «из коробки» в **прозовом** режиме: Claude Code читает `CLAUDE.md`, по числовой
команде открывает `roles/<роль>.md` и становится этой ролью. Это сознательно простой режим (см.
[architecture/agentic-workflows.md](architecture/agentic-workflows.md)).

Если хочется, чтобы правила держались не только на дисциплине агента, в `.claude/` лежит
**опциональная** исполняемая обвязка. Её можно не копировать — прозовый режим работает без неё.

- **`.claude/agents/`** — роли как настоящие субагенты с frontmatter. Tool-scoping делает прозовые
  запреты аппаратными: reviewer и auditor физически read-only; BA / QA-UAT / SA пишут только
  документы; architect без Edit/Bash на проде. Сюда же входит адверсивная панель (`red-team`,
  `blue-team`, `arbiter` — см. [core/adversarial-panel.md](core/adversarial-panel.md)) и дормантный
  `auditor` (диспатчится ad-hoc без цифры, как панель).
- **`.claude/skills/`** — авто-подтяг **знаний** (Agent Skills). Это тонкие триггеры с `description`,
  которые агент сам подтягивает по релевантности и которые указывают на канонические файлы
  `core/stack/architecture/domain` (без дублирования). Универсальные `debugging` / `code-quality` /
  `memory` / `spec-driven` идут в пакете; под выбранные stack/arch/domain скиллы эмитит генератор. Роли (субагенты)
  и панель (`/panel`) — не скиллы.
- **`.claude/commands/role.md`** — slash-команда `/role R D T`: парсит числа, резолвит роль по
  таблице из `CLAUDE.md`, запускает субагента.
- **`.claude/commands/panel.md`** — slash-команда `/panel <решение>`: прогоняет адверсивную панель
  (red → blue → arbiter + codex) и доводит до ADR.
- **`.claude/hooks/` + `settings.json.example`** — хуки-гейты: `check-loc.sh` (warn-сигнал по
  размеру отредактированного файла — будит SRP-суждение, не блокирует), `checkpoint-precompact.sh`
  (recovery-чекпойнт перед сжатием контекста), `check-no-todo.sh` (opt-in Stop-хук конституции) +
  опциональный Stop-хук на тесты.

Включение и детали — [.claude/README.md](.claude/README.md).

## Эволюция шаблонов

Эти шаблоны — живые. По мере работы с разными проектами вы будете замечать:

- правила, которых не хватает (добавляйте);
- правила, которые over-specialized под ушедший контекст (удаляйте);

## Словарь терминов

- **Регламент** — слой правил, ролей и команд, который этот пакет накладывает на проект. Не код, а
  то, как агент с кодом работает.
- **Роль** — режим работы агента с собственными правами и обязанностями (архитектор, разработчик,
  ревьюер и т.д.). Запускается числовой командой или по имени.
- **`R D T`** — числовая команда «роль R берёт задачу T из гайда дня D». Основной режим работы.
- **День** — инкремент плана проекта; его задачи лежат в `docs/day-<N>-guide.md`.
- **PROJECT-STATE** (`docs/PROJECT-STATE.md`) — снимок текущего состояния проекта, который архитектор
  читает на входе в день. **Снимок, не журнал:** обновляется перезаписью, решённое выкидывается;
  история «что было» остаётся в git, решения «почему» — в ADR.
- **ADR** (Architecture Decision Record) — запись об одном архитектурном решении: контекст, решение,
  последствия, отклонённые альтернативы.
- **Адверсивная (состязательная) панель** — процедура разбора важного решения: red-team бьёт,
  blue-team защищает, arbiter выносит вердикт. Запуск — `/panel`.
- **C+ / spec-driven** — цикл «контракт-первый» для задач с жёстким контрактом (парсеры,
  вычисления, валидаторы): тест до кода, независимый автор тестов, состязательная вычитка тестов.
- **Гейт** — условие, без выполнения которого работа не считается завершённой. Бывают **mechanical**
  (объективно проверяются командой: зелёные тесты, чистая сборка) и **accountability** (суждение
  агента под запись: слои, scope, пороги размера файлов). Размер файла — сигнал, будящий
  SRP-суждение, а не блокирующий лимит.
- **Owned-долг (версионный долг)** — обязательство сопровождать сохранённый артефакт по мере его
  устаревания. Пакет сознательно его избегает (см. [ADR-001](docs/adr/001-scope-process-overlay.md)).
- **Дормантная роль** — роль, метод которой записан, но цифры в `roles.json` у неё нет; доступна
  ad-hoc, активация цифрой — под гейтом реального опыта.
- **Прозовый режим** — работа без исполняемой обвязки `.claude/`: агент просто читает markdown и
  становится ролью.
